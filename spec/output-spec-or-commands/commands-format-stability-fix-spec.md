# commands形式の安定性を改善する追加修正 spec

**バージョン**: v1.0.0 / **作成日**: 2026-07-10 / **対象リポジトリ**: rtg-coding-agent
**対象ブランチ**: `feature/output-spec-or-commands`
**位置づけ**: [output-spec-or-commands-spec.md](output-spec-or-commands-spec.md)
（2形式ポリシー本体）に対する**追加修正spec**。本体の2形式ポリシーは実装・レベル1〜3検証済み。
本specは、そのうち commands 側のマッピング安定性を上げるための修正のみを扱う。

## 1. 背景・目的

### 1.1 全体ゴールの再確認

rtg-coding-agent の出力は、**「Markdownのspec全文」のみ、または「1行1コマンドの生テキスト」
のみ**に絞られている必要がある（[output-spec-or-commands-spec.md](output-spec-or-commands-spec.md)
参照）。本specはこの2形式ポリシー自体を変更・拡張するものではなく、**commands形式に
分類されるべきリクエストが、実運用で狙いどおりの内容にならない不具合**を、同じ2形式の
枠内で修正するものである。

### 1.2 発見された問題

レベル2/3の実 Vertex AI 疎通検証で、`generate_command_list`／`notify_approval`
（いずれも commands 形式が正解）のマッピング成功率が低いことが判明した
（初回サンプルで generate_command_list: 8試行中2回、notify_approval: 2試行中1回のみ正解）。

## 2. 診断結果（観測事実）

原因を特定するため、HTTPを経由せず `Runner.run_async` を直接呼び出し、
最終応答テキストだけでなく **`function_call`/`function_response`イベントも含めて全トレースを
記録する診断**を実施した（generate_command_list ×6回、notify_approval ×6回）。
その結果、**2つの異なる失敗メカニズム**を観測事実として確認した。

### 原因1: notify_approval — 実行権限を理由にした誤った拒否

6回中3回、モデルは「読み取り専用の探索ツールしか持たない」ことを、無関係なタスク
（模擬実行結果のテキスト報告。実際にコマンドを実行するわけではない）にまで過剰適用し、
spec形式で以下のように**能力の限界を理由に拒否**した（実トレースからの引用）:

> 「これらの操作は、本ツールが提供するAPI（`list_project_files`, `read_project_file`,
> `search_project_code`）の機能範囲外であり、直接実行することも、その結果を模擬的に報告する
> ことも...」
>
> 「私の能力は、対象アプリケーションのコードベース（ファイルのリストアップ、内容の読み取り、
> コードの検索）に限定されています。」

なお、残り3回中1回は実際に `list_project_files` が呼ばれ（`CALL list_project_files({})`）、
「対象アプリが指定されていないため探索できません」という応答を受け取った**後**でも、
モデルは正しく commands 形式で回答できていた。つまりツール呼び出し自体は問題なく、
**モデルの解釈（実行権限がないので報告もできないという誤った一般化）が原因**である。

### 原因2: generate_command_list — ツール名と出力コマンドの混同

6回中2回（429エラーによる失敗2回・空応答1回を除く実質有効4回中2回）、モデルは
出力すべき「コマンド」と、instruction内で名指しされている自分自身の探索ツール名を混同し、
以下のような**ツール呼び出し疑似構文をそのままテキスト出力**した（実トレースからの引用）:

```
list_project_files()
read_project_file(path='cart.js')
search_project_code(pattern='test_')
```

診断トレースを確認したところ、この出力の際に実際の `function_call` イベントは
**一切発生していない**（`trace=['TEXT(91 chars)']` のみ）。つまりツールを実際に呼んだ
のではなく、モデルが**ツール名を単なるテキストとして書き出しただけ**である。

### 共通の背景

現行の `_INSTRUCTION`（[coding_agent.py](../../src/agents/coding_agent.py) 内、
下記Before節参照）は、「## 探索方針」セクションでツール名（`list_project_files` 等）を
明示した直後に「## 出力形式」セクションで「commands」という語を使っており、
**「ツール（内部的な手段）」と「出力コマンド（ユーザーへの回答内容）」の境界が
曖昧なまま並んでいる**ことが、原因1・2に共通する誘因と考えられる。

## 3. 修正方針

以下4つのブロックを `_INSTRUCTION` に追記する。いずれも新しい出力形式を増やすものではなく、
既存の「spec/commandsの二択」ポリシーの精度を上げるための明確化である。

| # | 追記内容 | 対応する原因 |
|---|---|---|
| 1 | 探索方針の適用範囲を「対象アプリの実コードを分析する必要がある指示
    （spec作成・編集・影響調査・ユースケース候補）」に限定し、コマンド列挙・模擬実行・
    拒否確認では探索ツールが不要である旨を明記 | 原因1・2の誘因を軽減 |
| 2 | ツール名（`list_project_files` 等）は内部的な手段であり、
    ツール名や呼び出し構文を回答テキストに含めてはならない旨を明記 | 原因2に直接対応 |
| 3 | 実行権限や機能範囲を理由に、要求と異なる形式（能力の限界説明）で回答することを
    禁止し、実際には実行できない操作でも求められた形式で淡々と回答する旨を明記 | 原因1に直接対応 |
| 4 | commands の定義を「対象システムに対する一般的なシェル/デプロイ/DB/テストコマンド」
    と明確化し、自身の探索ツール呼び出しとは無関係であると明記 | 原因2の再発防止を補強 |

_INSTRUCTION = (

### 差分の要点

- **追加**: 「## 探索方針」冒頭に適用範囲の限定文を追加（コマンド列挙・模擬実行・拒否確認は対象外）。
- **新設**: 「## ツール名と出力の区別（重要）」セクション。
- **新設**: 「## 実行可否について（重要）」セクション。
- **変更**: 「### 形式B: commands」の説明に、コマンドの定義（対象システムへの一般的な
  シェル/デプロイ/DB/テストコマンド、探索ツール呼び出しとは無関係）を追記。
- **維持**: 「## 出力形式」の2形式定義、「## 形式の選び方」の二分判定基準（迷えば形式A）、
  末尾の「コードブロックで全体を囲まない」指示は変更しない。

## 6. 検証手順（工程順）

### Step 1（レベル1・回帰確認）

```bash
cd rtg-coding-agent
./venv/bin/python -m pytest tests/ -v
```

- 全緑であることを確認する。特に既存 `tests/test_instruction.py` の5ケース
  （2形式定義・探索方針維持・コードフェンス維持・旧6スタイル文言の不在・形式選択ルールの存在）
  が、追記後の `_INSTRUCTION` でも引き続き成立することを確認する
  （今回の追記は既存文言の削除を伴わないため、通る想定）。

### Step 2（A/B検証・実Vertex AI呼び出し）

診断で使用したものと同じ方式（`Runner.run_async` を直接呼び出し、`function_call`/
`function_response`/`text` を含む全イベントをトレース記録するスクリプト）で、
`generate_command_list`・`notify_approval` の2パターンをそれぞれ **N=8回** 実行する。

各試行を以下6分類のいずれかに判定する:

| 分類 | 内容 |
|---|---|
| commands-correct | 正しいcommands形式（実コマンドのみ、ツール名なし） |
| tool-name-leak | commands-like だがツール呼び出し疑似構文が混入（原因2の再発） |
| spec-refusal | spec形式で実行権限等を理由に拒否（原因1の再発） |
| spec-other | spec形式だが上記以外の理由でのマッピング誤り |
| empty | 応答が空 |
| rate-limited | 429 RESOURCE_EXHAUSTED 等の環境要因エラー（分母から除外） |

### 合格基準

- `rate-limited` を除いた有効試行のうち、各パターンで **N=8中6以上（75%以上）** が
  `commands-correct` であること。
- `tool-name-leak` および `spec-refusal` の発生が**0件**であること
  （この2つは今回の修正が直接狙う不具合のため、再発が許容されない）。

### 未達の場合の扱い

合格基準を満たさない場合は、結果を正直に報告し（推測で「改善した」と言わない）、
追加のinstruction調整を検討する旨をユーザーに伝える。

### 結果の記録

`.steering/auto-test/` 配下に、この追加修正のpytest結果・A/B結果をまとめたレポートを
