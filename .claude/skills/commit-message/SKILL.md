# Git コミットメッセージ作成スキル

出典: Chris Beams 著 *How to Write a Git Commit Message* (`https://cbea.ms/git-commit/`)  
目的: `git log`・`shortlog`・`blame`・`revert`・`rebase` などを効果的に使えるよう、一貫性が高くコンテキスト（「なぜ」）が保存されたコミットメッセージを作成する。

---

## 基本原則

1. **diff は「何が」変わったかを示す。コミットメッセージは「なぜ」変えたかを説明する。** 長期的な価値はコンテキストを素早く再確認できること。
2. **一貫性は独創性に勝る。** チーム規約があれば迷いがなくなり、読みやすい履歴ができる。
3. **標準化すべき3つの次元**:
   - **スタイル**: マークアップ、折り返し幅、文法、大文字化、句読点
   - **内容**: 本文に何を書くか（書かないか）
   - **メタデータ**: Issue ID・PR番号・参照の記法

---

## 7つのルール（必須）

1. **サブジェクトと本文を空行で区切る**
2. **サブジェクトは50文字以内**（上限72文字）
3. **サブジェクトの先頭は大文字**
4. **サブジェクトの末尾にピリオドを付けない**
5. **サブジェクトは命令形で書く**
6. **本文は72文字で折り返す**
7. **本文は「何を・なぜ」を書く（「どうやって」は書かない）**

### テンプレート

```text
<命令形・先頭大文字・50文字以内・末尾ピリオドなし>

<本文: 何が変わったか・なぜ変えたか。72文字で折り返す。段落は空行で区切る>

<箇条書きも可。72文字で折り返す>

Resolves: #123
See also: #456, #789
```

## ワークフロー

### 0) 事前確認・安全チェック

以下を実行する:

```bash
git status --porcelain
git branch --show-current
git diff --staged --stat
git diff --staged
git log --oneline -10
```

ルール:
- ステージ済みの変更がなければ **推測でコミットしない**。何をステージするか明確なら追加し、不明なら確認する。
- diff に秘密鍵・トークンが含まれている場合は **停止してユーザーに警告する**。

### 1) 「なぜ」と「何が」を diff から読み取る

`git diff --staged` から以下を抽出する:

- **何が変わったか**: 実装手順ではなく、変更後の状態を最小かつ正確に表現する
- **なぜ変えたか**: 動機・バグ・要件・運用上の課題・制約
- **変更前 vs 変更後**: 以前の振る舞いと現在の振る舞い
- **副作用・リスク**: マイグレーション・互換性への影響・データ形式の変更・パフォーマンスへの懸念

まとめるのが難しければ、アトミックなコミットへの分割を提案する。

### 2) サブジェクト行を作成する（最初に作る）

**必須要件**
- 命令形。**動詞で始める**（"Add", "Fix", "Remove", "Refactor", "Update", "Prevent", "Document", "Revert" など）
- 先頭の文字を大文字にする
- 末尾にピリオドを付けない
- **50文字以内**（絶対上限72文字）

**命令形チェック**
サブジェクトは以下の文章を完成させる必要がある:

> "このコミットを適用すると … <サブジェクト> される"

OK の例:
- "Refactor subsystem X for readability"
- "Update getting started documentation"
- "Remove deprecated methods"

NG の例:
- "Fixed bug …"（過去形）
- "Changing behavior …"（現在進行形）
- "More fixes …"（曖昧）

### 3) 本文が必要か判断する

変更が自明でトリビアルな場合（例: タイポ修正）は**1行**のコミットで十分。

**本文が必要なケース**:
- 変更が非自明
- 変更の理由が diff から明らかでない
- 振る舞いの変化・トレードオフ・副作用がある
- 制約・意図・今後の課題を残したい

### 4) 本文を書く（何が・なぜ。どうやってではない）

**ルール**
- 本文の各行を**72文字で折り返す**（手動改行）
- 必要に応じて完全な文章を使う
- 段落は空行で区切る
- 箇条書きも可。読みやすく72文字で折り返す

**本文に書くべき内容**
- 問題の説明・動機
- 以前の振る舞いが不十分だった理由
- ユーザー・システム視点で何が変わったか
- 重要な副作用や非自明な影響
- 必要に応じてフォローアップの注記

**書かないこと**
- 「どうやって」の手順（diff に書いてある）
- コードコメントに書くべき実装の詳細

### 5) メタデータを追加する（任意・統一して）

参照はメッセージの**末尾**に置く:

```text
Resolves: #123
See also: #456, #789
```

### 5b) JIRA ID の解決

以下の優先順位で JIRA ID を決定する:

1. スキルが引数付きで呼ばれた場合（`$ARGUMENTS`）、それを JIRA ID として使う
2. なければ、現在のブランチ名から JIRA 形式のパターン（例: `PROJ-123`、`ABC-42`）を探して最初のマッチを使う
3. どちらでも見つからない場合、`Resolves:` 行を**含めない**

JIRA ID が見つかった場合、コミットメッセージの末尾に空行を挟んで追加する:

```
サブジェクト行

本文（あれば）

Resolves: PROJ-123
```

### 6) コミット前にバリデーションする

作成したメッセージを以下で確認する:

- サブジェクトが50文字以内（上限72文字）
- 命令形で「このコミットを適用すると…」テストに通る
- 先頭大文字・末尾ピリオドなし
- 本文がある場合、サブジェクトとの間に空行がある
- 本文が72文字で折り返されている
- 本文が「何が・なぜ」を説明し「どうやって」を避けている

### 7) ファイルに書き出す（必ず実行）

提案するコミットメッセージを `/tmp/commit-message.txt` に書き出し、クリップボードにコピーする:

```bash
cat > /tmp/commit-message.txt << 'EOF'
Fix parser handling of nested objects

ネストされたオブジェクトリテラルを条件式内で評価する際に
誤った結果を返していた問題を修正。
EOF
pbcopy < /tmp/commit-message.txt
```

### 8) コミットを実行する

ユーザーの承認後のみ `git commit -F` を実行する:

```bash
git commit -F /tmp/commit-message.txt
```

1行のサブジェクトのみの場合は `git commit -m "<subject>"` も可。複数段落の場合は `-m` を使わない。

---

## 出力の形式

呼び出されたときに出力すること:

1. `/tmp/commit-message.txt` に提案するコミットメッセージを書き出す
2. `pbcopy < /tmp/commit-message.txt` でクリップボードにコピーする
3. 以下を表示する:
   - コミットメッセージ全文
   - サブジェクト行の文字数
   - ファイルパス: `/tmp/commit-message.txt`
   - クリップボードへのコピー完了の確認
   - 実行コマンド: `git commit -F /tmp/commit-message.txt`

---

## 重要な指針

- 50文字以内でサブジェクトをまとめられないなら、そのコミットはアトミックではないかもしれない
- 「なぜ」は diff には書かれていないことが多い。今のうちに残す
- コミットメッセージは永続的なドキュメントとして扱う。未来の自分もステークホルダーの一人
