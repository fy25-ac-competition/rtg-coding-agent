# 別 Cloud Run アプリ（呼び出し元）から本エージェントを呼ぶための必須事項

`rtg-coding-agent` は `--no-allow-unauthenticated`（IAM 認証必須）で
Cloud Run にデプロイされている。同一 Google Cloud プロジェクト内であっても、
Cloud Run 間の呼び出しは**自動的には認証されない**ため、
呼び出し元アプリ（RTG 本体など）を Cloud Run にデプロイする際は
以下を必ず設定すること。

## 対象サービス情報

| 項目 | 値 |
|---|---|
| プロジェクト | `reverse-turing-demo-suzuki` |
| リージョン | `asia-northeast1` |
| サービス名 | `rtg-coding-agent` |
| Service URL | `https://rtg-coding-agent-947565277742.asia-northeast1.run.app` |

## 必須事項チェックリスト

- [ ] 呼び出し元アプリを**専用のサービスアカウント**で Cloud Run にデプロイする（Compute Engine のデフォルト SA を使い回さない）。
- [ ] その呼び出し元 SA に、本サービス（`rtg-coding-agent`）への `roles/run.invoker` を付与する。
- [ ] 呼び出し元コードから、**audience = 本サービスの URL** とした ID トークンを取得し、`Authorization: Bearer <id-token>` ヘッダーを付けてリクエストする。
- [ ] 呼び出し元の環境変数 `A2A_CODING_AGENT_URL` に本サービスの URL を設定する。
- [ ] （推奨）呼び出し元も同じ `asia-northeast1` にデプロイし、レイテンシを抑える。

## 手順

### 1. 呼び出し元用のサービスアカウントを作成

```bash
gcloud iam service-accounts create <呼び出し元アプリ名>-sa \
  --display-name="<呼び出し元アプリ名> SA" \
  --project=reverse-turing-demo-suzuki
```

### 2. 呼び出し元 SA に本サービスへの `run.invoker` を付与

同一プロジェクトでも、Cloud Run はサービス単位で IAM 認可が必要。
これを付与しないと呼び出し元からのリクエストはすべて **403** になる。

```bash
gcloud run services add-iam-policy-binding rtg-coding-agent \
  --region=asia-northeast1 \
  --member="serviceAccount:<呼び出し元アプリ名>-sa@reverse-turing-demo-suzuki.iam.gserviceaccount.com" \
  --role="roles/run.invoker" \
  --project=reverse-turing-demo-suzuki
```

### 3. 呼び出し元アプリを、そのサービスアカウントを指定して Cloud Run にデプロイ

```bash
gcloud run deploy <呼び出し元サービス名> \
  --image=<呼び出し元イメージ> \
  --region=asia-northeast1 \
  --service-account=<呼び出し元アプリ名>-sa@reverse-turing-demo-suzuki.iam.gserviceaccount.com \
  --set-env-vars="A2A_CODING_AGENT_URL=https://rtg-coding-agent-947565277742.asia-northeast1.run.app" \
  --project=reverse-turing-demo-suzuki
```

### 4. 呼び出し元コードで ID トークンを取得してリクエストする

Cloud Run 上では、サービスに紐づく SA が ADC（Application Default Credentials）として
自動的に使われる。**audience は必ず呼び出し先サービスの URL** にすること
（別サービスの URL を audience にした ID トークンは受理されない）。

**Python の例（`google-auth` ライブラリ使用）:**

```python
import google.auth.transport.requests
import google.oauth2.id_token
import httpx

TARGET_URL = "https://rtg-coding-agent-947565277742.asia-northeast1.run.app"

def get_id_token(audience: str) -> str:
    auth_req = google.auth.transport.requests.Request()
    return google.oauth2.id_token.fetch_id_token(auth_req, audience)

async def call_coding_agent(payload: dict) -> dict:
    token = get_id_token(TARGET_URL)
    async with httpx.AsyncClient() as client:
        resp = await client.post(
            TARGET_URL + "/",
            json=payload,
            headers={"Authorization": f"Bearer {token}"},
            timeout=120.0,
        )
        resp.raise_for_status()
        return resp.json()
```

> `google-auth` がインストールされていない場合は `pip install google-auth` を追加する。
> ローカル開発時（Cloud Run 上でない場合）は、事前に
> `gcloud auth application-default login` を実行しておけば同じコードで動作する。

**curl での動作確認例（デプロイ後の疎通テスト用）:**

```bash
TOKEN=$(gcloud auth print-identity-token --audiences=https://rtg-coding-agent-947565277742.asia-northeast1.run.app)
curl -s -X POST "https://rtg-coding-agent-947565277742.asia-northeast1.run.app/" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","id":1,"method":"message/send","params":{"message":{"role":"user","parts":[{"text":"test"}]}}}'
```

## よくある失敗

| 症状 | 原因 | 対処 |
|---|---|---|
| 呼び出し元から常に 403 | 呼び出し元 SA に `run.invoker` が付与されていない | 手順2を実施 |
| 403（`run.invoker` は付与済み） | ID トークンの audience が本サービス URL と一致していない | `fetch_id_token` / `print-identity-token --audiences` の値を確認 |
| ローカル実行時のみ失敗 | ADC が未設定 | `gcloud auth application-default login` を実行 |
| デフォルト SA で動かしている | Compute Engine のデフォルト SA は権限管理が粗くなりやすい | 呼び出し元専用 SA を作成して使う（手順1） |
