## 2. 事例本体（入力プロンプト全文 → 出力全文）

### create_spec-1

**入力プロンプト**:
```
以下の要望を実現する spec ファイル（Markdown・実現方針/変更範囲/手順）を作成してください:
カート画面にクーポン機能を追加したい
```

**出力全文**（spec形式）:
```markdown
## 実現方針
カート画面にクーポン機能を追加するため、以下の要素を実装します。

1.  **クーポン管理機能の追加**: クーポンコード、割引種別（固定額/割合）、割引額、有効期限、利用条件などを管理する機能。
2.  **カート画面のUI変更**: クーポンコード入力欄と適用ボタン、および適用されたクーポンの表示、割引後の合計金額の表示を追加します。
3.  **カート情報の更新ロジックの追加**: クーポンが適用された際のカート内の商品合計金額、送料、税金、最終的な支払い金額の再計算ロジックを実装します。
4.  **APIの追加/更新**: クーポン適用/解除のためのAPIエンドポイントを準備し、カート情報取得APIにクーポン情報を追加します。
5.  **データベーススキーマの変更**: クーポン情報を保存するためのテーブルと、カートまたは注文に適用されたクーポンを紐付けるためのテーブルを追加します。

## 変更範囲
*   **データベース**:
    *   `coupons` テーブルの新規作成（クーポンコード、割引種別、割引額、有効期限、ステータスなど）。
    *   `cart_items` または `carts` テーブルへのクーポン関連フィールド追加、または `cart_coupons` 中間テーブルの新規作成（カートとクーポンの紐付け）。
*   **バックエンドAPI**:
    *   `POST /api/cart/coupon` (クーポン適用): クーポンコードを受け取り、カートに適用し、更新されたカート情報を返す。
    *   `DELETE /api/cart/coupon` (クーポン解除): カートからクーポンを解除し、更新されたカート情報を返す。
    *   `GET /api/cart`: カート情報に適用されているクーポン情報と、割引後の金額を含めるように変更。
*   **ビジネスロジック**:
    *   クーポンの有効性（有効期限、利用回数、適用条件）を検証するロジック。
    *   クーポンの割引をカートの合計金額に適用し、最終的な支払い金額を計算するロジック。
*   **フロントエンド（カート画面）**:
    *   クーポン入力フォーム (`<input type="text">` および `<button>`) の追加。
    *   適用されたクーポン（コード、割引額）の表示エリア。
    *   割引前の合計金額、割引額、割引後の合計金額を表示するエリアの更新。
    *   クーポン適用/解除APIを呼び出すJavaScriptロジック。
    *   APIからのレスポンスに基づきUIを更新するロジック。
*   **テスト**:
    *   クーポン適用、解除、無効なクーポンコード、期限切れクーポンなど、各シナリオの単体テストおよび結合テストの追加。

## 手順
1.  **データベーススキーマの定義と実装**:
    *   `coupons` テーブル（`id`, `code`, `discount_type` [e.g., 'fixed', 'percentage'], `discount_value`, `min_order_amount`, `max_discount_amount`, `expiry_date`, `usage_limit`, `used_count`, `created_at`, `updated_at`）。
    *   `cart_coupons` テーブル（`id`, `cart_id`, `coupon_id`, `applied_discount_amount`, `created_at`）または`carts`テーブルに`coupon_id`および`discount_amount`フィールドを追加。
    *   マイグレーションスクリプトを作成し、データベーススキーマを更新します。
2.  **バックエンドのビジネスロジックの実装**:
    *   クーポンコードの検証ロジック（存在確認、有効期限、利用条件、使用制限など）。
    *   カートの合計金額に対するクーポンの割引額計算ロジック。
    *   カートにクーポンを適用/解除するサービス層のロジック。
3.  **APIエンドポイントの実装**:
    *   `POST /api/cart/coupon` の実装。リクエストボディからクーポンコードを受け取り、カートに適用し、更新されたカート情報をJSONで返します。
    *   `DELETE /api/cart/coupon` の実装。カートからクーポンを解除し、更新されたカート情報をJSONで返します。
    *   `GET /api/cart` のレスポンスに、適用中のクーポン情報と、クーポン適用後の合計金額を追加します。
4.  **フロントエンドのUIコンポーネントの実装**:
    *   カート画面にクーポンコード入力フォームと「適用」ボタン、および「解除」ボタン（クーポン適用時のみ表示）を追加します。
    *   クーポン適用後の割引額と最終合計金額を表示する部分を実装します。
    *   クーポン適用/解除時にAPIを呼び出し、レスポンスに応じてUIを更新するJavaScriptロジックを実装します。
    *   ローディング表示やエラーメッセージ表示などのUXも考慮します。
5.  **テストの実装**:
    *   新しいAPIエンドポイントとビジネスロジックに対する単体テストおよび結合テストを作成します。
    *   フロントエンドのE2Eテストにクーポン適用/解除のシナリオを追加します。
6.  **デプロイと監視**:
    *   本番環境へのデプロイと、新機能の動作監視を設定します。
```

*   **データベース**:
    *   商品テーブルとカテゴリテーブル間のリレーションシップ（外部キー制約など）が適切に定義されていることを確認します。必要であれば、関連付けを追加・修正します。

    4.  カテゴリIDが指定されなかった場合は、すべての商品を返すようにします。

    8.  ローディングスピナーやエラーメッセージなど、ユーザー体験を向上させるためのUIフィードバックを実装します。

### investigate_impact-1

**入力プロンプト**:
```
以下のコマンドが対象システムに与える影響を、実現方針/変更範囲/手順の枠組みで報告してください。
--- コマンド ---
ALTER TABLE orders ADD COLUMN coupon_id INTEGER;
```

**出力全文**（spec形式・影響調査を3見出しに押し込めた形）:
```markdown
## 実現方針
`ALTER TABLE orders ADD COLUMN coupon_id INTEGER;` コマンドは、データベース内の `orders` テーブルに `coupon_id` という名前の新しい整数型カラムを追加します。このカラムは、既存の行に対してはデフォルトで NULL 値が設定されます。

## 変更範囲
*   **データベーススキーマ:** `orders` テーブルの定義が変更され、`coupon_id` カラムが追加されます。
*   **データ:** 既存の `orders` テーブルの全行において、新しく追加された `coupon_id` カラムは NULL 値を持ちます。
*   **アプリケーションコード:** `orders` テーブルを参照する既存のアプリケーションコードは、この新しいカラムの追加自体によって即座にエラーになることはありませんが、`SELECT *` を使用している場合や、カラムの順序に依存している場合には影響を受ける可能性があります。新しく追加された `coupon_id` を利用するためには、アプリケーションコードの修正が必要となります。
*   **パフォーマンス/可用性:** テーブルのスキーマ変更は、テーブルのロックを引き起こす可能性があり、大規模なテーブルの場合、コマンド実行中は `orders` テーブルへのアクセスが一時的にブロックされるか、性能が低下する可能性があります。

## 手順
1.  **バックアップの取得:** `ALTER TABLE` コマンドを実行する前に、`orders` テーブル、またはデータベース全体のバックアップを取得します。これは、予期せぬ問題が発生した場合にデータを回復できるようにするためです。
2.  **本番環境外でのテスト:** 開発環境やステージング環境でこの変更を事前にテストし、アプリケーションへの影響やパフォーマンスへの影響を確認します。
3.  **コマンドの実行:** 適切なデータベースクライアントまたは管理ツールを使用して、以下の SQL コマンドを実行します。
    ```sql
    ALTER TABLE orders ADD COLUMN coupon_id INTEGER;
    ```
4.  **変更の検証:** コマンド実行後、`DESCRIBE orders;` または同等のスキーマ確認コマンドを実行して、`coupon_id` カラムが正しく追加されたことを確認します。また、既存のデータに対して `coupon_id` が NULL となっていることを確認します。
5.  **アプリケーションの更新（任意）:** `coupon_id` カラムを利用するアプリケーションロジックが必要な場合、該当するアプリケーションコードを更新し、デプロイします。
```

**観察**: 「影響調査」という本来spec形式とは異なる意味の依頼が、`## 実現方針`＝コマンドの意味説明、`## 変更範囲`＝影響を受ける対象、`## 手順`＝安全な実行手順、という自然な読み替えで3見出しに収まっている。2形式強制ポリシーの狙い通りの結果。

## 3. 観測まとめ（次アクションの参考情報）

### 3.1 2形式不変条件は完全に成立（10/10）

`other` 判定・ツール名リーク・実行権限を理由にした拒否は1件も観測されず、既存の2形式ポリシー・commands安定化修正が実運用でも機能していることが確認できた。

### 3.2 「形式は合っているが内容が薄い/不自然」なケースが3件

- `generate_command_list-1`: commands形式だが `edit`/`refine` という非実在コマンドで実用性が低い（同種spec・同メソッドの `-2` は実在コマンドで良好）。
- `notify_rejection-1`: spec形式だが「変更範囲: なし」等、実質的な情報がほぼ無い。
- `suggest_usecases-1`: spec形式に押し込められた結果、候補列挙という本来の依頼内容が `## 実現方針` に圧縮され、他2見出しが空疎な一般論になっている。

これらはいずれも「2形式のいずれかに強制する」という設計そのものが意図してトレードオフしている点であり（[output-spec-or-commands-spec.md §2.3](../../spec/output-spec-or-commands/output-spec-or-commands-spec.md#23-全リクエストを受理し2形式へ強制マッピング) に既述）、本カタログはその副作用を実出力で裏付けたものと言える。改善の要否はこのレポートの範囲外とし、事実として記録する。

### 3.3 同一メソッド・類似specでも出力の粒度にばらつきがある

`generate_command_list-1` と `-2` は同じ「specからコマンド化」という依頼でありながら、一方は擬似コマンド、他方は実行可能な git/pytest/npm コマンド列という差が出た。1回試行のみのため統計的な傾向とまでは言えないが、代表事例として記録しておく。
